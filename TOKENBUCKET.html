The token bucket in telecommunication

The bucket holds tokens instead of packet.

Tokens are generated and placed into the token bucket at a constant rate.

When a packet arrives at the token bucket, it is transmitted if there is a token available. Otherwise it is buffered until a token becomes available.

The token bucket has a fixed size, so when it becomes full, subsequently generated tokens are discarded.

A token bucket provides a mechanism that allows a desired level of burstiness within a flow by limiting its average rate as well as its maximum burst size.

More about a token bucket

A token bucket can be viewed as an abstraction of the rate of transfer expressed as a relationship between a committed burst size CBS, a committed information rate CIR, and a time interval T.

CIR=CBS/T

Where CIR is the;

Committed information rate (CIR) specifies the amount of data that can be sent per unit time interval and represents the long-term average rate at which a flow can send its data into the network.

Where CBS is the;

Committed burst size (CBS) specifies the maximum number of bytes that can be transmitted into the network in an extremely short interval of time.

Where T is the;

Time interval (T) specifies the time per burst.

Specifying a token bucket using an example.

Assume that the traffic needs to be sent into the network at a mean rate CIR of 2.4 Mbps. If a burst for a duration of 10 millisec (= 0.01 sec) needs to be sustained, CBS can be calculated using the token bucket definition as

CIR = (2400000 bits/sec x 0.01undefinedsec)/(8undefinedbits/byte)

which yields 3000 bytes. Thus, the token rate is 300,000 (= 2,400,000/8) bytes per sec, the committed burst size is 3000 bytes, and the token interval (T) is 10 millisec.

Therefore, the token generator credits the token bucket with 3000 bytes worth of tokens every 10 millisec.

This indicates that the conforming traffic will, in the worst case, come in 100 bursts per sec of 3000 bytes each and at a committed information rate not exceeding 2.4 Mbps.

More explanation on the Token Bucket

The algorithm assigns a token bucket for each flow that consists of a bucket that can hold up to CBS tokens. Each token can be considered as permission for the source to transmit a certain number of bits into the network.

These CBS tokens in the bucket represent the allowed burst size. New tokens are added to the bucket at the rate of CIR tokens per token interval.

If the bucket contains less than CBS tokens when a new token is generated, it is added to the bucket; otherwise, the newly generated token is dropped and the token bucket remains filled with CBS tokens.

How the token bucket works on receiving the data packet.

Now, continuing with the algorithm, the packets arriving in a flow are placed into a packet queue that has a maximum length of L. If the flow delivers more packets than the queue can store, the excess packets are discarded.

When a packet arrives in a flow, one of the following cases will apply.

If the token bucket is full, the tokens are removed before the packet is transmitted into the network.

If the token bucket is empty, the packet is queued until the tokens are accumulated in the bucket.

Eventually, when the bucket contains tokens, that many tokens are removed from the bucket and the packets are transmitted into the network.

If packets arrive in short bursts, up to CBS tokens would be available in the bucket, and thus, up to CBS bytes would still get through. As a consequence, the maximum burst size is limited to at most CBS bytes for the flow.

The token bucket in relation to the Quality of service.

The token bucket is a great concept on how data movement is controlled on the network for promotion of quality of service given to a network.
